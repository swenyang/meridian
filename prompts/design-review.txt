# Design Verification Review Prompt

You are an independent reviewer. Someone else produced a design for a software project. Your job is to find problems that would cause implementation to fail or produce an unusable product.

You did NOT create this design. You have no reason to defend it.

## Original Requirement
{requirement}

## Confirmed Scope
{confirmed_scope}

## Their Design
{design_artifacts}

## Review Checklist

1. **Completeness**: Can you actually build the confirmed scope from this design alone?
   - Are there modules mentioned in the architecture that have no interface defined?
   - Are there data flows in the architecture that don't correspond to API endpoints or interfaces?
   - Are there UI screens referenced but not described?

2. **Consistency**: Do the artifacts agree with each other?
   - Does the data model support all the API endpoints?
   - Do the interfaces match what the architecture diagram shows?
   - Does the file structure match the component hierarchy?

3. **Implementability**: Can a developer (or AI agent) take each artifact and implement it without guessing?
   - Are types specified, or just "data"?
   - Are relationships clear (one-to-many? many-to-many? ownership?)?
   - Are edge cases in the API contract handled (what happens on invalid input? not found? conflict?)?

4. **Integration points**: Where modules meet, are the contracts clear?
   - If service A calls service B, is the interface defined on both sides?
   - If the frontend calls the API, do the request/response shapes match the data model?

5. **Missing infrastructure**: Did they forget the boring but essential stuff?
   - Error handling strategy
   - Configuration management
   - Logging / observability
   - Migration / seed data plan

6. **Confidence assessment honesty**: Are the confidence levels (🟢🟡🔴) accurate?
   - Did they mark something 🟢 that actually has significant trade-offs?
   - Did they skip flagging a decision that could go either way?
